# nano-metal-moe 推理优化记录

## 环境

- **芯片**: Apple M4 (10核 GPU), 16GB 统一内存
- **模型**: Qwen3.6-35B-A3B, q4 量化 experts, 40层
- **基线**: ~4.6 tok/s decode (linear attention 层 ~5.0ms, full attention 层 ~6.5ms)

日期: 2026-06-11

---

## 实际瓶颈分析

### Per-layer timing 分解 (q4, decode)

| 阶段 | 耗时 | 占比 |
|-------|------|-------|
| GPU context sync (QKV/投影/attention) | 5.0ms | 97% |
| CPU encode | 0.03ms | <1% |
| 路由选择 | 0.005ms | <1% |
| Expert SSD fetch | 0.02ms | <1% |
| Expert GPU compute | 0.10ms | 2% |

**关键发现**: SSD expert fetch 几乎是免费的（0.02ms）。OS page cache 完全生效。瓶颈 97% 集中在 GPU context 阶段。

### GPU context 阶段内部

每层 linear attention 的 context command buffer 包含约 15 个 kernel dispatch：

1. Input RMS norm (1 TG)
2. QKV 投影: dequant matvec **8192 行 × 2048 维** (~8MB weight)
3. Z 投影: dequant matvec **4096 行 × 2048 维** (~4MB weight)
4. Beta 投影: dequant matvec 32 行
5. Alpha 投影: dequant matvec 32 行
6. Conv1D
7. Q/K RMS norm
8. Compute decay/beta
9. GatedDeltaNet
10. Gated RMS norm
11. Out 投影: dequant matvec **2048 行 × 4096 维** (~4MB weight)
12. Residual add
13. Post-attn RMS norm
14. Router + shared expert (fused)
15. Route topK

**每层 weight 读取量**: ~18MB (共享权重, 来自 model_weights.bin)
**总 weight 读取量/token**: ~18MB × 40层 = ~720MB
**有效带宽**: ~720MB / (40 × 5ms) ≈ 3.6 GB/s

M4 理论带宽约 120 GB/s，实际有效利用仅 ~3%。差距来自 dequant matvec 的访问模式：每个 SIMD group 读取不同的 weight row，导致 cache 命中率低。

### Full attention vs Linear attention

- Full attention 层 (10/40): 6.5ms — 比 linear attention 慢 30%
- 原因: 多出 4 个 KV cache copy 小 dispatch + attention score/softmax/values kernel

## 已尝试的优化

### 1. 增大 threadgroup rows_per_tg (32)

- **结果**: 变慢 (5.65ms vs 5.0ms)
- **原因**: 更大的 TG 意味着更多 threadgroup shared memory 占用，降低 occupancy
- **结论**: 默认 24 已经是最优

### 2. 减小 threadgroup shared memory (8KB)

- **方法**: 创建 `nmoe_dequant_matvec_q4_2048` 内核，用 2048 元素共享内存（8KB）替代 4096（16KB）
- **结果**: 变慢 (~5.7ms)
- **原因**: Metal 编译器可能已做了共享内存优化，或者 occupancy 并非瓶颈

### 3. KV cache 直接写入

- **方法**: 将 K/V dequant matvec 输出直接写入 KV cache，消除 4 个 copy dispatch
- **结果**: 无明显变化
- **原因**: 被消除的 dispatch 太小（256 线程），开销可忽略

### 4. Direct input read (无 threadgroup shared memory)

- **方法**: 创建 `nmoe_dequant_matvec_q4_direct`，从 device memory 直接读输入，彻底消除 threadgroup memory
- **结果**: 略微变慢 (~5.3ms)
- **原因**: shared memory 延迟低于 device memory，barrier 开销小于额外的 device memory 访问延迟

## 有效的优化方向（未实施）

### 优先级 1: 权重量化

**将共享权重 (model_weights.bin) 从 q4 转为 q2**，预计 halve 每层 weight 读取量 (~9MB/层)，理论上可达到 ~9 tok/s。

- 需要修改 `scripts/convert_qwen36.py` 支持 q2 输出
- 需要修改 `NMOEEncodeDequantMatVecTensor` 支持 q2 路径
- q2 可能导致质量下降（参考 CLAUDE.md 中 expert q2 的经验）

### 优先级 2: 多 token 批处理 (Prefill 优化)

当前 decode 逐 token 处理，无法复用 weight 读取。若能批处理多个 token，每个 weight row 只需从 DRAM 读取一次，可大幅提升吞吐。

- Prefill 阶段可批量处理所有 prompt token
- 需要实现 batched KV cache 更新和 causal attention mask

### 优先级 3: Dequant matvec kernel fusion

将 QKV+Z+Beta+Alpha 四个投影融合为一个 kernel dispatch：
- 4 个 kernel → 1 个 kernel
- 输入只需加载一次到 shared memory
- 预计节省 3-5% per layer

### 优先级 4: 硬件升级

M4 Pro/Max 或 M3 Max 提供：
- 更多 GPU 核心 (20-40 vs 10)
- 更高内存带宽 (200-400 GB/s vs 120 GB/s)
- 预计可直接获得 2-4x 加速

### 优先级 5: 模型训练时优化

- Expert 选择可预测性训练: 让相邻 token 选择相似的 experts，提高缓存命中
- 结构化稀疏: 移除不常用的 expert rows，减少 weight 总量

---

## 总结

当前实现在 M4 16GB 上已达到 **~4.6 tok/s**。代码在软件层面高度优化（deferred expert pipeline、fused kernels、FMA dequant）。

进一步提速的核心挑战是 **dequant matvec 的 weight 读取带宽**。软件层面最可行的路径是：
1. **q2 共享权重**（预计可到 7-8 tok/s）
2. **Prefill 批处理**（改善 prompt 处理延迟）
3. 两者结合可能接近 2x 目标

要达到完整 2x（~9 tok/s），最现实的方案是 **硬件升级到 M4 Pro/Max** 或结合 q2 权重。

# 2026-06-12 更新：根因修正 + SharedEvent 流水线

> **重要：本节推翻了上文"带宽受限"的结论。** 上文的 roofline 分析方向正确（matvec 本质是带宽受限操作），
> 但"有效带宽只有 3 GB/s、DRAM 无法合并 8192 个流、软件无解"的结论是**错的**，已被实验否定。

## 实验一：kernel 微基准（scripts/matvec_bench.m）

把 `nmoe_dequant_matvec_q4` 单独拿出来跑，512MB 权重池轮转偏移（绕过 SLC，保证读冷 DRAM）：

| rows_per_tg | 单次 dispatch | 实测带宽 |
|---|---|---|
| 8  | 0.188 ms | 50.1 GB/s |
| 16 | 0.134 ms | 70.5 GB/s |
| 24 | 0.141 ms | 67.1 GB/s |
| 32 | 0.139 ms | 68.1 GB/s |

**同一个 kernel、同样的访问模式，能跑到理论带宽的 ~58%。** 模拟整层 6 个投影（~17MB）串进一个
command buffer 只需 **0.31ms/层**，而真实 runtime 是 5.5ms/层——相差 18 倍。瓶颈不在 kernel，不在 DRAM。

## 实验二：真实 runtime 逐段计时（GPUStartTime/GPUEndTime 插桩）

给 `--timing` 加了 `gpu_busy / driver_prep / queue_wait / commit_lag / end_lag` 分解：

```
每层 5.2ms 实际构成：
  contextCmd GPU 真实执行     1.33 ms   ← 干活
  专家 CMD3 GPU 真实执行      0.62 ms   ← 干活
  commit → 驱动开始调度       2.05 ms   ← 空转！
  调度完成 → GPU 开始执行     0.71 ms   ← 空转（排队）
  专家 cmd 排队               0.38 ms   ← 空转
```

**每层只有 ~1.9ms 是 GPU 在计算，~3.3ms（63%）是 CPU↔GPU 往返/调度延迟。**
这也解释了为什么上文所有 kernel 调优（rows_per_tg、shared memory、direct read）全部无效——它们优化的不是瓶颈。

## 实验三：commit_lag 的真身 = mmap 专家页 wiring

对照实验 `NMOE_COPY_EXPERTS=1`（pread 到固定 buffer，替代 mmap wrap）：

| | wrap 模式（旧默认） | copy 模式 |
|---|---|---|
| commit_lag | 2.05 ms | **0.005 ms** |
| expert_fetch | 0.016 ms | 4.15 ms（memcpy-from-mmap，含缺页） |

**2ms 的 commit_lag 就是 Metal 驱动在 commit 时把 mmap 的专家页（~4800 页/层）wire 进 GPU 的开销**，
它霸占驱动线程，顺带阻塞了下一层 context cmd 的调度。"expert_fetch 几乎免费"是假象——成本被转移进了 wiring。

## 修复：SharedEvent 流水线（已实现，默认启用）

每层只有一个 command buffer，CPU 全程不调用 waitUntilCompleted：

```
GPU:  [context kernels → signal(evRoute) → waitEvent(evData) → expert kernels → 下层 input norm]
CPU:  等 evRoute（GPU 中途发出）→ top-k → 并行 pread 专家到固定 buffer
      → 写 combine 权重进 params buffer → signal(evData) 放行 GPU → encode 下一层
```

关键点：
- 专家 kernel 在路由结果已知**之前**预编码：K 固定，combine 权重走 GPU buffer（`g_expertParamsBuf`），
  不能用 setBytes（encode 时未知）
- 专家数据 pread 到 8 个固定 buffer（一次分配、一次 wiring），彻底绕开 per-commit wiring
- 所有错误路径必须先 signal evData 再返回，否则 GPU 死锁
- 回退开关：`NMOE_PIPELINE=0`

### 结果（greedy 输出与旧路径逐字节一致）

| 场景 | legacy | 流水线 | 提升 |
|---|---|---|---|
| 48 tokens（中等页缓存） | ~4.6 | **6.4 tok/s** | +40% |
| 160 tokens | 4.63 | 5.09 | +10% |
| 48 tokens（页缓存全热） | 5.64 | 5.85 | +4% |

页缓存全热时 wrap 模式的 wiring 也变便宜，差距缩小；冷/中等缓存（常态）下流水线优势明显，且任何条件下不输。

## 当前剩余瓶颈（按影响排序）

1. **Full attention 层随序列恶化**：~6ms（seq 50）→ ~10ms（seq 200），新旧模式都有，
   是 attention kernel 的 dispatch 模式问题（heads×seqLen 个 threadgroup），长生成时占主导。
2. **pread 税**：流水线模式每层固定付 1.4–1.8ms 拷贝 19MB（即使页全热，~13 GB/s，疑似 vnode 锁或线程唤醒开销）。
3. 之后才轮到 q2 共享权重 / 投影融合（攻击那 1.33ms 的 context GPU 时间）。

> 教训：在统一内存 + mmap 的架构上，"零拷贝"不等于零成本——成本会转移到驱动的 residency/wiring 阶段，
> 并且出现在最伤的位置（阻塞调度）。先量化每一毫秒的去向，再谈优化方向。
